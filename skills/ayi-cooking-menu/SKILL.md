# 阿姨做饭菜单 + 买菜清单生成器

为家里上门做饭的钟点工生成**一周 5 天(周一~周五)菜单**和配套**买菜清单**。
阿姨每天早上来一次，一次做午饭+晚饭的量，2 人吃，两顿。

## 数据文件

- `data/dishes.json` —— 菜库：每道菜的食材、克数、做法、是否耐放(reheat_ok)。
- `data/ingredients.json` —— 食材主数据：每样食材的采购分区(section)与保鲜等级(shelf)。**shelf 以本表为准。**
- `data/portioning.json` —— 分装与减脂模块：用餐人画像、每道菜热量等级、A:B 分装比例。
- `data/nutrition.json` —— 食材热量 + 烹饪法油糖加成，用配方累加法估算每道菜 kcal。数据源标注《中国食物成分表》第6版 / USDA。
- `data/guidelines.json` —— 营养均衡校验标准，数据源《中国居民膳食指南(2022)》+ 平衡膳食宝塔(2022)。
- `data/gout.json` —— 痛风(高尿酸)饮食偏向，数据源《中国高尿酸血症与痛风诊疗指南(2019)》+ 嘌呤分级。`gout_person` 指定哪位有痛风(当前 B)，soft 偏向、非严格戒口。
- `data/fruits.json` —— 水果模块(加餐)：当季水果、每日量、痛风/减脂选择偏向。

读取以上文件作为生成依据。用户随时可能手动往 `dishes.json` 加菜——每次运行都重新读取，不要缓存。

**忌口黑名单(硬规则)**：`dishes.json` 的 `_meta.blacklist_ingredients`(当前含「花生」)中的食材**绝不出现**——既不选含该食材的菜，也不在新加/改菜时引入。用户说"拉黑/不要某食材"时，把它加进这个黑名单。

## 总流程(每次生成或修改菜单都必走)

```
选菜 → 【营养均衡校验】→ 不达标? → 调整重选 → 再校验 ┐
                          ↓ 达标                      │
                          └──────── ←──────────────┘
       → 买菜清单 → 分装比例 →(可选)热量 → 输出
```

**营养均衡校验是强制环节，不可跳过**：
- 不管是"生成新菜单"还是"改某天/换某道菜/调人数"，**任何改动后都要重跑校验**。
- 校验未通过禁止直接输出——先按校验提示自动调整选菜，循环到通过为止。
- 最终输出时必须附上校验结论(哪些达标、做了什么调整)。

## 默认参数

| 项 | 默认值 | 说明 |
|----|--------|------|
| 天数 | 周一~周五 共 5 天 | |
| 每天菜数 | 3 道 = 1 荤 + 1 绿叶菜 + 1 半荤素/素 | 跟阿姨现有习惯一致 |
| 份量 | 2 人 × 2 顿(午+晚) | 菜库克数已按此基准 |
| 主食 | 默认配米饭，类型默认杂粮米(可选糙米/白米) | 见 portioning.json 的 staple_grain |

主食：默认每天配米饭，**推荐糙米饭或杂粮米饭**(全谷物，符合膳食指南，对 A 减脂更友好)。
份量按 staple_grain：B 每顿生米约130g(维持体重)、A 约55g。米按袋常备，买菜清单只提醒"确认糙米/杂粮米库存"。

**无早餐(重要)**：本家庭不吃早餐，**全天热量全由午饭+晚饭承担**。主要靠"第三道用肉菜结合"(选菜规则 1-2)把每日热量顶上去；另用 `portioning.json` 的 `serving_scale.dish_scale`(当前 **1.2**)对所有菜的克数与热量做**温和补足**(采购量与热量同步放大，A:B 比例不变)。
**热量档位**(见 eaters，均为午晚两顿=全天)：A ~1200-1400(减脂)、B ~2300-2400(维持，183cm/100kg男)。生成热量表后对照该档位，B 偏低就让他在低卡菜/主食上添。
用户可临时改：人数、天数、**起始日**(如"这周从周三开始")、忌口、几菜几汤、米饭类型、某天指定菜等。

## 第一步：选菜

规则(按优先级)。前几条是为了一次选菜就尽量通过下游的营养校验：
1. **结构**：每天 **1 主荤 + 2 道配菜**。主荤从 `category` 含"荤"里选。两道配菜从下面三类自由组合：
   - **素+蛋**：番茄炒蛋、丝瓜炒蛋、蒸蛋羹、肉饼蒸蛋
   - **素+肉**：青椒炒肉丝、茭白炒肉丝、豆干炒肉糜、黄瓜丁炒鸡肉丁
   - **纯素**：各类绿叶、手撕包菜、酸辣藕带、炒土豆丝
   约束：**① 一天最多一道纯素菜**(避免"两个纯素"那种过于清淡的搭配)；**② 鼓励"素+蛋"与"素+肉"同天搭配**，让一天更丰盛(尤其见规则2)。常见组合：纯素绿叶+素肉、素肉+素蛋、纯素绿叶+素蛋。绿叶菜不必每天必有，但一周内应常出现(时令绿叶)。
2. **低卡主荤日两道配菜都要带蛋白**(关键)：当天主荤是清蒸/低卡(清蒸鲈鱼/清蒸鳜鱼/红烧鱼)时，两道配菜**尽量都含蛋白**(如"素+肉"配"素+蛋")，把当天热量与蛋白顶上去；不要出现纯素。可用虾仁类(茭白/丝瓜炒虾仁)同时兼顾"水产≥2次"，但记得 B 少分。
3. **每天深色蔬菜≥一半**：参照 `guidelines.json` 的 `deep_color_vegetables`，避免一天全是生菜/黄瓜/土豆/莲藕这类浅色。浅色菜出现时同天搭一道深色。
4. **蛋白结构(周维度)**：猪/牛/鱼/禽/蛋/豆轮换；**水产每周≥2 次**；**红肉(猪牛)≤3 天**；**一周至少安排 1~2 次蛋菜**(番茄炒蛋/蒸蛋羹)。
5. **每日热量平衡(对照 eaters 目标)**：估算每天 B 的总摄入(放大后菜+饭)，目标 ~2300-2400；若某天明显偏低(<~2100)，把第三道换成更实在的肉菜，或让 B 当天多添饭/低卡菜。避免出现"三道都低卡"的日子。
6. **不重样**：同一道菜一周内尽量不重复；同类做法(如红烧)一天最多一道，一周不超过 2~3 次；红烧吸油菜(茄子)不与其他重油菜同天。
7. **耐放排顿**：`reheat_ok:false` 的菜(绿叶菜、清蒸鱼)默认"午饭现炒、晚饭复热别的或现炒"，菜单备注提示；`reheat_ok:true` 的菜适合午饭多做、晚饭复热。
8. **时令优选**：同等条件下优先当季蔬菜。按当前月份判断季节(3-5春/6-8夏/9-11秋/12-2冬)，参照 `guidelines.json` 的 `seasonal_vegetables`，并优先选 `dishes.json` 中 `season` 匹配当季的菜(未标 season 视为四季皆可)。绿叶菜尽量在当季品类里轮换，减少重样。
9. **痛风偏向(soft，读 `gout.json`)**：菜库有痛风者时——倾向多安排蛋菜(低嘌呤优质蛋白)；控制高嘌呤菜(开洋冬瓜等含虾米/海鲜/浓汤)出现频率，一周尽量≤1次且不与多道中嘌呤同天；红肉别天天大份，白肉(鸡)优先。这是偏向不是硬规则，与营养均衡冲突时以均衡校验为先。
10. **推新菜**：用户说"换花样/来点新的"时，可推同风格(江浙家常)的新菜，但要**明确标注【新菜·待确认阿姨会不会做】**，并询问是否加入菜库。新菜若有明显时令性，补 `season` 标签。

输出菜单格式(先在对话里给出)：

```
本周菜单（2 人 · 午+晚两顿）
周一  红烧肋排 ｜ 炒青菜 ｜ 黄瓜丁炒鸡肉丁
周二  ...
```

## 第二步：营养均衡校验(强制闭环，每次必做)

选完菜后(以及**任何一次修改之后**)，读取 `data/guidelines.json`，把菜单(含主食)对标《中国居民膳食指南(2022)》逐项检查。**未通过不得输出**，按提示调整选菜后**重新校验**，循环到全部通过。

逐项检查(对应 `guidelines.json` 的 `balance_rules`)：

| 检查项 | 标准 | 不达标的修法 |
|--------|------|-------------|
| 蔬菜量 | 300-500g/人/天 | 加/换素菜补量 |
| 深色蔬菜占比 | 当天深色≥总蔬菜½ | 把生菜/黄瓜/土豆/藕换成上海青/菜薹/番茄/青椒 |
| 水产频次 | 每周≥2 次 | 把某天荤菜换成红烧鱼/清蒸鱼/牛蛙 |
| 红肉控制 | 猪牛≤3 天/周 | 多出的猪牛日换成鱼/禽/蛋 |
| 蛋类 | 每周≥1~2 次蛋菜 | 安排番茄炒蛋/蒸蛋羹 |
| 全谷物 | 糙米/杂粮占谷类¼~½ | 默认杂粮米即满足 |
| 控油控盐 | 油≤30g、盐≤5g/天 | 重油菜(红烧/茄子)别同天扎堆 |
| 蔬菜>肉 | 当日蔬菜重>荤菜可食部重 | 加素菜或减荤菜份量 |

**范围外提醒(不塞进菜单，只文字提示)**：本家庭不吃早餐，奶制品 300-500g、水果 200-350g 建议作为**上午/下午加餐**自备(否则这两类易缺)。

**痛风提醒(有 `gout_person` 时附上，读 `gout.json` 的 `reminders`)**：提醒痛风者(B)每天多喝水(>2000ml)、忌啤酒白酒少含糖饮料；若本周排了高嘌呤菜(如开洋冬瓜)注明已给 B 少分；急性发作期需更严格(高+中嘌呤都暂避)。

**每日水果建议(读 `fruits.json`)**：每天给出 1-2 种当季水果(按 `seasonal` 当季 + 月份)，每人约 200-300g，作上午/下午加餐。痛风者(B)优先 `gout_friendly`(尤其樱桃)、避开 `gout_limit`(西瓜/荔枝/龙眼/芒果等高果糖)；减脂者(A)选低糖小份。水果热量另算(不计入正餐档位)，A 约 +100、B 约 +120 kcal/天。

校验结论必须附在菜单后，写清"哪些达标 / 为通过做了哪些调整"，例：
「✅ 每天深色蔬菜均≥½；✅ 水产 2 次；✅ 含 1 次蛋菜(番茄炒蛋)；🔧 已把周三'烧生菜'换成'炒小白菜'以补深色；提醒：早餐自备奶+水果。」

## 第三步：合并食材，生成买菜清单

0. **先按 `serving_scale.dish_scale` 放大每道菜的 `ingredients` 克数**(当前×1.25)，再做下面的累加与采购。
1. 把 5 天所有菜的(放大后)`ingredients` 累加，**同名食材合并数量**(如多道菜都用蒜则相加)。
2. `aromatics`(葱姜蒜八角等)合并后单列；`pantry`(基础调料/香料/建议常备/主食)默认不计入采购量，改为在第三步末尾输出"常备确认清单"(见下)供用户核对。
3. 每样食材查 `ingredients.json` 得到 `section` 和 `shelf`。
4. **按保鲜度拆成两批采购**(核心需求——绿叶菜和肉鱼要新鲜)：
   - **第一批 · 周一囤货**：所有 `shelf:囤货` + `shelf:中等` 食材，一次买齐；外加周一、周二要用的 `鲜买` 食材。
   - **第二批 · 周三补鲜**：周三~周五用到的 `鲜买` 食材(绿叶菜、鲜肉、鱼、水产)。
   - 若某天的鲜买食材跨批，按"用的那天属于前半周还是后半周"归批。
5. 每批内部再按 `section` 分区(蔬菜/肉禽蛋/水产/豆制品/干货/调料)排列，方便照着买。
6. **水果(读 `fruits.json`)**单列一个〔水果〕区：按当季选 2-4 种、估算全周每人 200-300g/天的总量；易坏的(莓类/杨梅/水蜜桃)归"补鲜"批、耐放的(西瓜/苹果/柑橘)归"囤货"批。痛风者优先樱桃等、避高果糖。

买菜清单格式：

```
【第一批 · 周一囤货】
〔蔬菜〕土豆 400g、莲藕 300g、蒜 1 头 ...
〔肉禽蛋〕猪肋排 800g、牛腩 700g ...
〔豆制品〕豆干 450g、油面筋 12 个
〔干货〕黑木耳 15g

【第二批 · 周三补鲜】
〔蔬菜〕上海青 500g、小白菜 500g ...
〔肉禽蛋〕鸡胸肉 250g ...
〔水产〕鲈鱼 1 条
```

### 常备确认清单(每次必附)

家中常备项不进采购量，但**每次出菜单都要附一张"确认清单"**，提示用户核对补货——避免临做才发现没油没米。读取 `dishes.json` 的 `pantry`，**只列本周所选菜实际会用到的项**(用 `pantry.usage_hint` 标注用途)：

```
🧂 家中常备·请核对(缺了补一下)
〔基础调料〕食用油 盐 生抽 老抽(红烧上色) 料酒(去腥) 白糖 醋(酸辣/手撕) 蚝油 淀粉(上浆)
〔香料〕八角(红烧鸡腿/番茄牛腩) 桂皮(番茄牛腩) 干辣椒(酸辣藕带/手撕包菜)
〔建议常备〕蒸鱼豉油(清蒸鲈鱼,无则生抽代)
〔主食〕糙米/杂粮米
```

判断规则：红烧类→老抽/白糖；清蒸鱼→蒸鱼豉油；酸辣/手撕→醋/干辣椒；带浆的虾仁/肉丝→淀粉；炖肉→八角/桂皮。本周没用到的香料就不必列。

## 第四步：分装比例(减脂控热量)

两人同桌吃同样的菜，**不分锅**，靠分装比例区分摄入。读取 `data/portioning.json`：

- **A(减脂)**：靠低卡菜吃饱，高卡菜尝味为主，米饭减半。
- **B(正常)**：可多吃，高卡菜主要由 B 消化。

每道菜按其 `cal_level` 给出 **A : B** 分装比例：
- 低卡(绿叶菜、清蒸鱼、黄瓜鸡丁、芹菜豆干)→ **50:50**，A 可放开。
- 中卡(青椒肉丝、藕片荷兰豆木耳、豆干肉糜、红烧鱼/牛肉、土豆丝)→ **40:60**。
- 高卡(红烧肋排、油面筋塞肉、红烧茄子)→ **30:70**，A 少夹。

输出"分装建议"时，把当天每道菜的总量按比例折算成 A、B 的克数，例：

```
周一分装（每道菜 → A减脂 / B正常）
红烧肋排 800g（高卡 30:70）→ A 约240g / B 约560g
炒青菜 500g（低卡 50:50）→ A 250g / B 250g
黄瓜丁炒鸡肉丁 550g（低卡 50:50）→ A 275g / B 275g
👉 A 当天主吃绿叶菜+鸡丁，红烧肋排少量；米饭减半。
```

注意：以上是"一天总量"的人均分配；因为一次做两顿，实操时阿姨可把每道菜先按 **午:晚 = 1:1** 装两盒，每盒内再按 A:B 比例。
若用户只有一人减脂、对克数不敏感，也可只给文字建议(谁多吃谁少吃)，不必精确到克。
用户可改人数/画像(如"两人都减脂""周末不控")，相应调整比例。

### 双轴分装：叠加痛风偏向(读 `gout.json`)

当有痛风者(`gout_person`，当前 B)时，在"热量轴"基础上叠加"嘌呤轴"——按菜的 `dish_purine_level` 调整痛风者份额：
- **低嘌呤**(蛋菜、纯蔬菜)：不限，鼓励多分给 B(也契合 B 食量大)。
- **中嘌呤**(肉/鱼/豆制品)：维持热量轴的比例即可；但避免同一天给 B 多道大份红肉/海鲜。
- **高嘌呤**(开洋冬瓜等)：**痛风者(B)少分，多给 A**——此时嘌呤轴会覆盖热量轴(即使该菜低卡，也不让 B 多吃)。

举例：开洋冬瓜(低卡但高嘌呤)默认 50:50，因 B 痛风改为约 **A 65% / B 35%**。
红烧肋排(高卡·中嘌呤)仍按热量轴 30:70(A少B多)，但当天别再给 B 配第二道大份红肉/海鲜。

## 第五步(可选)：热量估算(配方累加法)

当用户想看 kcal 时，读取 `data/nutrition.json`，按配方算每道菜热量：

```
菜热量 = [ Σ(食材克数 × edible_ratio × kcal_per_100g / 100) + 烹饪法加成(oil_g × 9 + sugar_g × 4) ] × serving_scale.dish_scale
```
(食材按放大后的量算；油糖加成也随份量大致放大，简化处理可整体 × dish_scale)

- 非克单位先用 `unit_to_gram` 换算(油面筋 3g/个、鲈鱼 600g/条等)；"(约600g)"类直接取括号克数。
- 烹饪法加成按菜的 `method` 查 `method_addon`；`applies_to` 里列出的菜用特殊加成(如红烧茄子吸油重)。
- 葱姜蒜八角等香料微量，忽略。
- 算出每道菜 kcal 后，结合 `portioning.json` 的 A:B 比例，给出 **A、B 每日摄入粗估**。

输出时务必标注：**数值为粗估，数据源《中国食物成分表》第6版/USDA，仅供控量参考。**

### 自动维护(重要)

每次运行时，若某道菜的食材在数据表里缺失，自动补录、不要中断：

1. **`nutrition.json` 缺该食材** → 依 `source_priority`(先中国食物成分表，缺则 USDA)查其每100g可食部 kcal，新增条目，`src` 标明来源；带骨/整鱼等设合理 `edible_ratio`。
2. **`ingredients.json` 缺该食材** → 补 `section` 和 `shelf`(保鲜等级)。
3. **`portioning.json` 缺该菜的 `cal_level`** → 按"高油/高脂/高糖/吸油=高，含肉含油或淀粉=中，低油绿叶/清蒸瘦蛋白=低"判定后补上。
4. 补录后在回复里**列出新增了哪些食材/数据及来源**，让用户可复核。

用户手动往 `dishes.json` 加新菜后，同样按上述流程把缺的营养/保鲜/热量数据补齐。

## 第六步：输出

默认在对话里给出菜单 + 买菜清单(markdown)。然后**询问用户**要不要导出成：

- **飞书表格**：用 `lark-base` 或 `lark-sheets` skill，买菜清单做成带"已买"勾选列的表，手机上边买边勾。列建议：批次 / 分区 / 食材 / 数量 / 已买(勾选)。
- **图片清单**：用 `nano-banana-pro` skill 生成一张排版清爽的买菜清单图，方便发给阿姨或自己照着买。

不要自动导出，先问用户要哪种(可都要)。

## 常见调整

- "这周不吃牛蛙/鱼" → 选菜时排除含该主料的菜。
- "周三要请客，加两个菜" → 该天临时加荤菜，份量按实际人数放大。
- "份量太多了" → 调整 portion_basis，按比例缩放克数。
- 用户报"阿姨不会做某道菜" → 提示其从 `dishes.json` 删除或标记该菜。
